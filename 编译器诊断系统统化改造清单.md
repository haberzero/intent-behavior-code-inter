# 编译器诊断系统统化改造清单

### **第一阶段：基础设施与协议升级 (The Plumbing)**
*   **[P0] 接口归一化**：
    - 修改 `core/compiler/support/diagnostics.py`，废弃 `DiagnosticSeverity` 枚举。
    - 让 `DiagnosticReporter` 接口直接引用 `core.domain.issue_atomic.Severity`。
    - 扩展接口方法：添加 `panic()`（对应 FATAL）和 `hint()`（对应 HINT）。
*   **[P0] IssueTracker 架构对齐**：
    - 重构 `core/support/diagnostics/issue_tracker.py`，使其内部存储的 `diagnostics` 列表由 Domain 层的 `Diagnostic` 实体组成。
    - **协议化位置决议**：移除基于反射的 `getattr` 逻辑，严格通过 `core.domain.issue.Locatable` 协议获取位置信息。
*   **[P1] 适配器清理**：
    - 移除 `core/compiler/support/issue_adapter.py` 中的严重程度映射逻辑（`_map_severity`），因为全系统现在共用同一套 Domain 枚举。

### **第二阶段：组件生产端重构 (The Producers)**
*   **[P1] Lexer & Scanner 改造**：
    - 将 Lexer 产生的原始错误字符串包装为 `Diagnostic` 对象。
    - 为常见的词法错误（如未闭合的字符串、非法字符）定义标准错误码（如 `LEX_001`）。
*   **[P2] Parser 组件深度整合**：
    - 修改 `BaseComponent._loc`，确保所有 AST 节点自动满足 `Locatable` 协议，以便直接传递给诊断器。
    - 在语法错误报告中引入 `hint` 字段（例如：在“缺少冒号”时提供“是否忘记了代码块开始？”的提示）。
*   **[P1] Semantic Analyzer 语义增强**：
    - 统一语义错误的报告格式。例如：类型不匹配不再只是 `message`，而应携带具体的错误码（如 `SEM_002`）和位置节点。

### **第三阶段：错误流控与抛出机制 (The Flow)**
*   **[P1] 异常回溯重构**：
    - 更新 `check_errors()`。当存在 ERROR 或 FATAL 时，不再抛出泛泛的 `Exception`，而是抛出带有完整诊断列表的 `core.domain.issue.CompilerError`。
*   **[P2] Scheduler 跨模块整合**：
    - 修改 `core/compiler/scheduler.py`，使其能够正确聚合来自不同模块（每个模块都有独立的 `IssueTracker`）的诊断信息，并生成一份全项目的编译报告。

### **第四阶段：质量与清理 (The Polish)**
*   **[P2] 诊断打印格式化**：
    - 在 `core/support/diagnostics/issue_tracker.py` 中实现符合 Domain 定义的 `Diagnostic` 打印逻辑，确保文件名、行号、列号格式在控制台输出中高度一致。
*   **[P3] 废弃代码移除**：
    - 彻底删除编译器内部残留的 `issue_adapter.py` 等“补丁”文件。
