# DynamicHost 动态宿主说明文档

> 本文档客观描述 DynamicHost（动态宿主）在 IBC-Inter 项目中的定义、作用和最小实现目标。
>
> **生成日期**：2026-03-21
> **版本**：V1.0

---

## 一、DynamicHost 是什么

### 1.1 基本定义

DynamicHost 是 IBC-Inter 运行时系统中的一个核心组件，负责**管理和协调子解释器的执行**。

在 IBC-Inter 的架构中，存在两种解释器角色：
- **主解释器（Main Interpreter）**：宿主进程的主要执行环境，负责加载和执行 IBC-Inter 代码
- **子解释器（Sub Interpreter）**：由主解释器通过 DynamicHost 启动的隔离执行环境

### 1.2 核心作用

DynamicHost 提供了以下核心能力：

| 能力 | 说明 |
|------|------|
| **隔离执行** | 在独立的环境中运行 IBC-Inter 代码，不干扰主解释器的状态 |
| **环境继承** | 子解释器可以继承主解释器的插件系统、UTS 类型系统、公理体系 |
| **状态恢复** | 通过快照机制保存和恢复执行状态，支持"返回到跳出点" |
| **信息交互** | 通过显式的文件读写机制与子解释器交换信息 |

### 1.3 与传统"沙盒"的区别

DynamicHost 不是传统意义上的沙盒或虚拟机：
- 不提供进程级隔离
- 不提供内存保护
- 不拦截系统调用

DynamicHost 提供的是**实例级隔离**，即在同一个 Python 进程中，通过独立的对象图谱和注册表实现逻辑隔离。

## 二、为什么需要 DynamicHost

### 2.1 IBC-Inter 的混合执行模型

IBC-Inter 是一种混合编程语言，同时包含：
- **确定性代码**：由解释器直接执行
- **非确定性行为**：通过 LLM 调用执行

DynamicHost 解决了以下问题：

| 问题 | DynamicHost 的解决方案 |
|------|----------------------|
| LLM 调用可能失败 | 子解释器独立执行，失败不影响主环境 |
| 代码需要动态生成 | 子解释器可以执行动态生成的代码 |
| 复杂流程需要分支执行 | 通过多个子解释器并行/串行执行不同分支 |
| 外部工具调用隔离 | 子解释器拥有独立的状态，不会污染主环境 |

### 2.2 与 AI Agent 的关系

在 IBC-Inter 的愿景中，DynamicHost 是构建 AI Agent 的基础设施：

```
┌─────────────────────────────────────────┐
│           主解释器 (Main)                │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │    AI Planner / Orchestrator     │   │
│  └─────────────────────────────────┘   │
│                  │                      │
│                  ▼                      │
│         ┌────────────────┐              │
│         │  DynamicHost   │              │
│         └────────────────┘              │
│           │           │                 │
│           ▼           ▼                 │
│    ┌──────────┐ ┌──────────┐           │
│    │子解释器A │ │子解释器B │           │
│    │(任务1)   │ │(任务2)   │           │
│    └──────────┘ └──────────┘           │
└─────────────────────────────────────────┘
```

AI Planner 决定执行策略，DynamicHost 负责创建和管理子解释器执行任务。

## 三、DynamicHost 的信息交互机制

### 3.1 显式文件交互原则

DynamicHost 与子解释器之间的信息交互遵循**显式文件交互原则**：

| 方式 | 说明 |
|------|------|
| **文件读写** | 通过 file 插件显式地进行硬盘读写 |
| **返回值** | 子解释器可以返回基本内置类型（int/str/bool/float/none） |
| **异常传播** | 子解释器的异常可以传播到主解释器 |
| **诊断持久化** | 子解释器的诊断信息可以保存到硬盘供主环境查阅 |

### 3.2 为什么不用隐式内存交互

| 隐式内存交互的问题 | 显式文件交互的优势 |
|-------------------|-------------------|
| 复杂且沉重 | 简单直接 |
| 难以调试和追踪 | 易于检查和验证 |
| 容易引入难以发现的bug | 状态可见可追溯 |
| 需要复杂的生命周期管理 | 无需额外管理 |

## 四、DynamicHost 最小实现目标

### 4.1 现阶段最小目标

| 目标 | 优先级 | 说明 |
|------|--------|------|
| **全隔离运行环境** | 必须 | 能够启动一个完全隔离的子解释器 |
| **继承插件/UTS/公理体系** | 必须 | 子解释器能够使用主解释器的所有插件和类型系统 |
| **不干扰主解释器** | 必须 | 子解释器的执行不会影响主解释器的状态 |
| **正确返回到跳出点** | 必须 | 子解释器执行完成后，主解释器能恢复到原位置继续执行 |
| **返回基本内置类型** | 高 | 子解释器可以返回 int/str/bool/float/none |
| **异常正确传播** | 高 | 子解释器的异常不会直接传播到主解释器，但主解释器可以用一些调试机制或文件交互机制来查看子解释器的记录 |

### 4.2 现阶段不实现的功能

以下功能被明确排除在当前阶段的最小实现之外：

| 功能 | 排除原因 |
|------|----------|
| 子解释器独立注册插件 | 当前阶段不允许 |
| 容器类型返回值 | 当前阶段禁止 |
| 进程级隔离 | 实例级隔离已足够 |
| 复杂权限控制 | 当前阶段不需要 |

### 4.3 未来扩展方向

DynamicHost 的未来扩展包括：

| 扩展 | 说明 |
|------|------|
| **运行时快照** | 保存和恢复完整运行状态 |
| **Breakpoint 接口** | 支持现场保存/恢复/回溯 |
| **复杂返回值** | 支持容器类型、插件类返回值 |
| **并发执行** | 多个子解释器并行执行 |
| **跨进程隔离** | 真正的进程级隔离 |

## 五、DynamicHost 在 IBC-Inter 中的定位

### 5.1 与解释器的关系

```
┌─────────────────────────────────────────┐
│              Engine                     │
│  (顶层编排，协调所有组件)                 │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│         Interpreter (解释器)             │
│  (执行 IBC-Inter 代码)                   │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│        DynamicHost (动态宿主)             │
│  (管理子解释器，提供隔离执行环境)         │
└─────────────────────────────────────────┘
```

Interpreter 是代码的执行者，DynamicHost 是子任务的管理者。

### 5.2 与插件系统的关系

DynamicHost 依赖插件系统提供服务：

| 插件 | DynamicHost 使用方式 |
|------|---------------------|
| **AI** | 启动的子解释器可以使用 AI 插件进行 LLM 调用 |
| **IDBG** | 子解释器可以使用 IDBG 插件输出调试信息 |
| **HOST** | DynamicHost 本身是 HOST 插件提供的功能 |
| **File** | DynamicHost 使用 File 插件进行显式文件交互 |

## 六、术语说明

| 术语 | 说明 |
|------|------|
| **主解释器** | 宿主进程的主要解释器实例，负责协调和调度 |
| **子解释器** | 由 DynamicHost 创建的隔离解释器实例 |
| **隔离执行** | 子解释器拥有独立的注册表和对象图谱 |
| **环境继承** | 子解释器可以继承主解释器的插件和类型系统 |
| **快照恢复** | 保存主解释器状态，执行子任务后恢复 |
| **返回到跳出点** | 子任务完成后，主解释器从原位置继续执行 |

---

## 七、设计原则总结

1. **实例隔离而非进程隔离**：通过独立的对象图谱和注册表实现逻辑隔离
2. **显式交互**：通过文件和返回值进行信息交换，不做隐式内存共享
3. **最小可用**：先实现最小功能集，确保稳定后再扩展
4. **AI Agent 基础设施**：为构建 AI Agent 提供执行环境支持
5. **不干扰主环境**：子解释器的执行不会影响主解释器的状态
