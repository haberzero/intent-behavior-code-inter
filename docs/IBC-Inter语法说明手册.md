# IBC-Inter 语法说明手册

交互式意图行为代码 —— IBC-Inter (Intent Behavior Code - Interactive)，是一种旨在连接确定性程序逻辑与非确定性自然语言推理模型的实验性混合编程语言。本手册旨在为开发者提供其最基本的设计理念说明，以及其最基本的语法索引。

## 1. 引言

### 1.1 技术背景

在传统的编程语言中，如果期望实现在代码中接入与 LLM 的交互，往往会涉及到繁琐的函数调用定义或者接口封装。为了提示词的可复用性，开发者有时不得不设计复杂的辅助函数来参与提示词文本的构建。而很多时候，这些操作本质上却只是在摆弄字符串，或者说得更好听一些 —— 构建系统化的提示词工程。

除此之外，现有的各类 Agent 自身也面临一些独特的困难和挑战。

以 Skills 技术栈为例，Skills.md 文件层层嵌套，试图用条件注入/多步注入的手段避免上下文污染，甚至出现了把 Skills.md 本身写得像是编程代码一般的局面，这实质上完全违背了 Markdown 的设计初衷 —— 供人阅读。而代价就是难以理解，可复用性相对糟糕，两个独立 Skills 仓库之间几乎永远不可能无缝兼容。想要复用就难免遇到本地适配问题和兼容性地狱。

“我的 Opencode 为什么不能调用我的Skills？调用了之后又为什么不听指令？”相关的问题在社区中可从来不少见。

类似的问题，在MCP，乃至更早的原始 Function Calling 的时代其实就已经出现。加了太多MCP，反而导致语言模型不知所措，长时间思考，烧了大量的不必要token，却最终获得一次“格式错误/调用失败”，这其中的沮丧和挫败感，相信相关的开发者都深有体会。

在现有的解决方案中，似乎更倾向于通过加强对模型的“工具调用能力”的强化训练来解决这种指令遵循/上下文污染的问题，背靠模型自身的性能以及训练算力的堆叠，这些问题在短期内确实仍然可以被粉饰。

但是，类似的解决方案并不利于 LLM 的平等技术普惠：对旗舰模型的依赖，对算力发展的依赖，只会愈发加重模型垄断、算力垄断，乃至 GPU 垄断，即便对于大公司来说，参与垄断竞争意味着员工会持续收获痛苦与高压 —— 哪怕是那些最优秀最顶尖的研发人员 —— 直到某一方把水抽干，熬到某个公司得到胜利，赢家通吃。差异化竞争在逐步失效，小公司的生存空间正在被极限挤压。

而于此同时，相关问题也意味着 Agent 构建对普通人的巨大门槛，非专业工程师几乎不可能独立构建出属于自己的绝对可控 Agent，导致普通人无法用普惠性的技术去高效、高性价比地解决自己的实际需求，只得求助于日益臃肿/沉重的 Agent 软件 —— 其中一部分甚至还在疯狂滥用 Vibe Coding ，忽视安全问题，把责任感抛诸脑后，对工程伦理视若无物，对全社会的计算机信息安全都造成了极端的损害。

### 1.2 IBC-Inter 的技术理念

针对上述所有这些问题，我尝试提出一个想法 —— 直接从编程语言这个层级作为切入点，尝试构建一个高效、可控、可复用，且不依赖于具体大模型的自然语言编程体系。设计上要简单易用，工作流程上要直观可控，要提供充分可靠的调试工具和监控工具，要让小尺寸语言模型也能在精确的工作流中展示自己的能力极限，让普通人也能无痛构建属于自己的最简 Agent。

这也就直接促成了交互式意图行为代码 (IBC-Inter) 的诞生，将 Intent（意图）、Behavior（行为）与 Code（代码）无缝连接，把提示词系统和 LLM 调用提升为编程语言中的一等公民，把 LLM 的行为视作表达式/函数调用：

- Code (代码)：确定性的骨架。负责数据结构定义、状态维护、文件交互、流程控制，保证程序的长期稳定性。
- Behavior (行为)：交互的桥梁。由 LLM 在运行时动态推理执行，并无缝接入代码逻辑。
- Intent (意图)：非确定性的上下文。作为环境的“背景信息栈”动态注入至提示词，告知 AI 当前逻辑运行的主要目的。
- Interactive (交互)：解释运行。LLM 的调用开销使得解释器设计不再主要专注于运行效率，而是去专注于更宏观的高级交互功能。

上述内容意味着 IBC-Inter 建议了一个 LLM 的边界：常规自然语言处理。我们不应强迫 LLM 进行其不擅长的精确计算，不应该试图让 LLM 用人类习惯的任务清单 (TODO List) 去进行长期流程控制和动态规划，更不应该试图让 LLM 从冗长晦涩的上下文包袱中提取出需要被使用的 tools 或 skills —— 更何况那些 skills 本身也是冗长的上下文包袱。

这些工作应交由确定性的代码脚本完成，通过显式地分解 LLM 的每一步工作职责，让它们能以最高效率发挥自己最大的作用。通过把沉重的上下文负担从 LLM 身上剥离，我们也可得以让长期被忽视的小尺寸开源模型也能够真正地参与到严肃的工作中。

## 2. 第一印象：Hello World

在 IBC-Inter 中，与 LLM 的交互需要从配置开始。核心 LLM 交互能力由API交互提供，你需要先完成显式的初始化。

现在让我们用一个简单的 hello world 代码来展示这一切。

```ibc-inter
# hello_world.ibci
import ai

# 配置 AI 引擎 (详情参考相关插件手册)
ai.set_config("https://your.base.url/v1", "your-api-key", "your-model")

print("Hello, IBC-Inter!")

# 使用Behavior语法，触发一个简单的LLM调用行为
str ai_msg = @~ 用热情洋溢的语气向世界打个招呼 ~
print(ai_msg)
```

在这个示例中：
- 第一行是标准的 Code 执行，调用内置的 `print` 函数。
- 第三行引入了 Behavior 语法 `@~ ... ~`。它会立即发起一次 LLM 调用，并将返回的结果赋值给字符串变量 `ai_msg`。

## 4. 意图注释与意图栈
意图注释是 IBC-Inter 的核心特性，它允许你动态地改变 AI 处理任务时的“心态”或“角色”。

### 4.1 常规单行意图

使用 `@` 符号定义意图，最基本的意图注释仅作用于紧随其后的下一行代码。

```ibc-inter
@ 你现在是一个冷酷的机器人
str response = @~ 回复用户的问候 ~
print(response)

@ 你现在是一个热情的助手
str response = @~ 回复用户的问候 ~
print(response)
```

### 4.2 意图修饰符

（TODO: 这部分功能其实还要明确，比如意图栈在函数调用阶段怎么作用，要提供一个明确的子函数意图栈继承机制。未来甚至应该把意图栈本身也接入对象系统，提供显式传递和控制的可能性，提供多重意图栈的管理可能性，然后把这些修饰符变成语法糖而不是核心语法？现阶段应该确实只有一个意图栈，用起来挺不方便的）

- @+ (叠加)：将意图压入栈，与现有上下文合并。
- @- (移除)：根据标签或内容从栈中移除特定意图。
- @! (覆盖)：忽略当前所有的全局和上层意图，仅以当前定义的意图为准。

```ibc-inter

str res = @~ 打个招呼 ~
print(result)

@-#1
@-#2
str result = @~ 列举一些简单的个体经营条款 ~
```

## 5. Behavior 与控制流

IBC-Inter 将 Behavior 与控制流深度融合，支持由 AI 判定逻辑走向。

### 5.1 条件分支
```ibc-inter
str input = "我今天心情很糟糕"

if @~ 判断这段话是否包含负面情绪：$input ~:
    print("检测到负面情绪")
else:
    print("情绪正常")
```

### 5.2 循环结构 (while/for)

IBC-Inter 提供了三种主要的循环方式：

- 标准 while 循环：基于布尔条件的重复执行。
- 标准 foreach 循环：遍历列表或字典。
- 意图驱动循环：使用 `for @~ ... ~` 语法，由 AI 判定是否继续循环。

(这里的范例书写有问题，for循环其实无法正确启动，需要改正，不是这样用的)

```ibc-inter
# 1. 标准 foreach
list items = [1, 2, 3]
for item in items:
    print((str)item)

# 2. 意图驱动循环 (Condition-driven loop)
# 只要 AI 判定结果为真，循环就会继续
for @~ 判定当前生成的文字是否足够生动？如果不够请继续优化 ~:
    str draft = @~ 优化这段文字：$draft ~
    if @~ 检查字数是否已超过 500 字 ~:
        break
```

## 6. LLM Function

IBC-Inter 用函数交互的视角来看待结构化 LLM 调用，并且提供了一种简单直观易用的结构化提示词书写方式

### 6.1 显式 LLM 函数

对于复杂的、多角色的提示词工程，IBC-Inter 提供了 `llm` 函数。它允许你显式划分系统指令和用户输入。

为了避免无谓的缩进块干扰提示词的书写/阅读，IBC-inter要求使用单独一行独自存在的 `llmend` 关键字来结束一个llm函数的定义。

这同时也是为了避免缩进块带来的歧义：缩进是应该作为空格文本处理还是应该作为对齐操作处理？引入llmend后，IBC-Inter就能允许llm函数内无歧义地直接书写对齐到llm关键字的提示词，允许作为空格字符存在的内容，从而避免过多的缩进带来开发时的心智负担。

对于复杂的 llm 函数，通常建议单独放置一个 `.ibci` 文件进行书写，抑或是始终考虑书写在源代码的末尾，从而保护核心逻辑的代码阅读体验。

## 7. 类系统与提示词协议

IBC-Inter 2.0 采用面向对象式设计，并强调具体对象在LLM交互过程中的行为定义。

### 7.1 基本的类定义与实例化

```ibc-inter
class MyUser:
    str name = "Anonymous"
    int age = 0
    
    func __init__(str n, int a):
        self.name = n
        self.age = a
        
    func greet() -> str:
        return "Hello, my name is " + self.name

User u = MyUser("Alice", 25)
```

### 7.2 提示词协议 (__to_prompt__)

这是 IBC-Inter 中，类系统与 LLM 进行交互时的主要桥梁。

当一个对象被作为插值变量被送入 `@~ ... ~` 或 `llm` 时，内核会自动调用该对象的 `__to_prompt__` 方法。

(TODO: 后续这部分内容需要更详尽的测试)

```ibc-inter
class Product:
    str id = "P001"
    str name = "测试用商品"
    str desc = "这个商品用于测试提示词协议的可用性"
    float price = 99.9
    
    # 提示词协议：对象被注入 Prompt 时自动触发
    func __to_prompt__() -> str:
        str prompt = 
            "商品编号：" + self.id + 
            "价格：" + (str)self.price +
            "商品名称：" + self.name +
            "商品描述：" + self.desc
        return prompt

Product p = Product()
# AI 看到的 $p 将是 "__to_prompt__" 返回的字符串
str ad = @~ 为这个商品编写一段通顺流畅的说明：$p ~

```

## 8. 异常自愈与 llmretry
(这个章节必须提前，要放到控制流章节之后)
在与 AI 交互时，不确定性是不可避免的。IBC-Inter 提供了专门的工具来处理这种不确定性。

### 8.1 llmretry 语法糖 (单行自愈)
[NEW 2.2] llmretry 提供了一种极其精简的重试引导。为了语法清晰，它必须**独立成行**书写，紧跟在可能触发模糊判定的意图语句之后。

```ibc-inter
int score = @~ 打分(0-100) ~ 
llmretry "请只返回数字"
```
注意：`llmretry` 会导致解释器精准返回到上一条语句的起始处重新评估，而非重启整个逻辑块。

### 8.2 llmexcept 机制 (逻辑自愈)
用于处理更复杂的判定失败或逻辑偏差。
```ibc-inter
try:
    for @~ 判定内容是否合规 ~:
        # 逻辑体...
llmexcept:
    # 告诉 AI 刚才哪里错了，并要求它重新判定
    retry "请不要给出模棱两可的回答，请只回答 是 或 否"
```

## 9. 系统限制与运行安全

为了保证在复杂意图嵌套下的稳定性，IBC-Inter 内置了一些物理限制：

- 递归深度限制：每一层意图包装或函数调用都会消耗一定的栈空间。系统会自动根据宿主环境调整 `max_call_stack`。（宿主机制还没讲，宿主机制以后应该提前说明）
- 指令数限制：默认 `max_instructions` 为 10,000 条，防止 AI 决策导致无限循环。(TODO 之后要提供更合理的机制，甚至引入定时器和看门狗之类的操作，现在这个限制太糟糕了)

### 10.2 插件架构说明

一个标准的 IES 2.0 插件由两个核心文件组成：
- _spec.py：定义插件导出的函数签名、类型信息及所需权限。
- __init__.py：具体的 Python 逻辑实现。

这种分离确保了内核可以在不加载（和运行）第三方代码的情况下，通过扫描 `_spec.py` 获知所有的语法能力。
