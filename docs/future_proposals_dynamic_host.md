# 未来提议：动态宿主 (Dynamic Host) 插件架构

本文件详细记录了关于 IBC-Inter 运行时扩展的一个重要提议。该提议旨在解决动态代码执行时的符号冲突问题，并提供一种高级的“热替换”机制。

## 1. 背景与问题
在当前的 Pool-Walking 架构中，编译器生成的 UID 是任务局部的。当运行时尝试动态加载并执行一段新代码（如通过 LLM 实时生成的片段）时，新旧代码池之间的 UID 可能会发生碰撞，且跨任务的符号合并逻辑过于沉重，会严重破坏编译器的解耦性。

## 2. 核心提议：动态宿主 (Dynamic Host)
提议引入一个名为“动态宿主”的高级插件，它不通过合并池子的方式来运行新代码，而是通过**逻辑栈**的方式进行上下文切换。

### 2.1 运行机制
1.  **快照 (Snapshot)**：插件暂停当前的解释器执行流，捕获当前的 `RuntimeContext`（变量状态）、`instruction_count`、`call_stack` 等现场信息。
2.  **压栈 (Push)**：将当前的 `artifact_dict`（包括旧的节点池、符号池等数据）压入一个逻辑栈中（可以是内存栈，也可以持久化到硬盘）。
3.  **热替换 (Hot Swap)**：将解释器的底层数据池（`node_pool`, `symbol_pool` 等）直接替换为新编译生成的动态代码池。
4.  **执行 (Execution)**：在全新的、独立的环境中执行动态代码，利用解释器现有的 Pool-Walking 能力。
5.  **出栈与恢复 (Pop & Restore)**：动态代码执行完成后，从栈中弹出旧的现场信息，将解释器状态完全还原到暂停前的时刻。

## 3. 技术优势
-   **物理隔离**：通过“时间上的隔离（栈）”完美避开了“空间上的碰撞（UID 冲突）”，无需实现复杂的重定位逻辑。
-   **非侵入性**：对内核的修改极小，只需在 `Interpreter` 中暴露必要的状态导出与热重载接口（反射机制）。
-   **灵活性**：支持嵌套调用（代码生成代码并再次压栈），为 AI 驱动的自我进化逻辑提供了基础设施。

## 4. 内核支持需求 (待实现)
为了支撑该插件，内核需要提供以下底层反射能力：
-   `Interpreter.save_state()` / `restore_state()`: 导出/导入解释器运行现场。
-   `Interpreter.hot_reload_pools(new_pools)`: 允许在不重新实例化解释器的情况下更换底层数据池。
-   `RuntimeContext.fork()`: 支持上下文的克隆与分支管理。
