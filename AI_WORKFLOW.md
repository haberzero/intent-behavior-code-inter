# AI 助手工作流程与规范

> 本文档总结与用户协作时建立的工作流程、规范和偏好。
> 供后续 AI 助手参考，确保工作一致性。

---

## 一、Git 工作流程

### 1.1 提交习惯

**频繁提交，小步快走**：
- 每完成一个逻辑单元立即提交
- 不累积大量修改后一次性提交
- commit message 使用中文，简洁描述完成内容

**提交消息格式**：
```
<类型> <简短描述>

<详细说明（可选）>
```

**示例**：
```
修复 MetadataRegistry 双轨制问题

- discover_all() 正确传递 registry.get_metadata_registry()
- host_interface.metadata 和 scheduler.registry 指向同一实例
```

### 1.2 分支策略

- 默认使用 `master` 分支
- 不创建功能分支（简化流程）
- 频繁同步远程：`git pull --rebase`（如需要）

### 1.3 状态检查

**每次重大修改后**：
```bash
git status
git log --oneline -5
```

**确认工作树干净**后再继续下一个任务。

## 二、Subagent 使用规范

### 2.1 何时使用

**使用场景**：
- 复杂的多文件搜索任务
- 需要并行分析不同模块
- 大规模代码审查

**不使用场景**：
- 简单查找（直接用 Grep）
- 单文件修改（直接 Read + Edit）
- 用户直接对话能解决的问题

### 2.2 使用方式

```
Task(tool): 
  description: "具体任务描述"
  query: "详细查询要求"
  response_language: "zh"  # 使用中文
  subagent_type: "search"   # 通常用 search 类型
```

### 2.3 结果处理

- Subagent 返回结果需要汇总分析
- 不要直接相信，必要时二次验证
- 关键发现要向用户汇报

## 三、用户对 AI 的工作要求

### 3.1 核心原则

**主动工作，不要等待**：
- 遇到问题主动分析，不要停下来等用户指示
- 可以做合理假设并继续，完成后向用户汇报
- 遇到无法解决的困难或偏离目标时，立即停止并汇报

**小步迭代**：
- 每次只做一件事或一小类相关的事
- 完成一个任务后再做下一个
- 不要试图一次性解决所有问题

**验证驱动**：
- 每次重要修改后运行 `git status` 确认
- commit 前检查变更范围是否合理
- commit 后检查 log 确认正确

### 3.2 汇报要求

**立即汇报的情况**：
1. 发现无法解决的困难问题
2. 发现会导致偏离原始目标的情况
3. 发现严重 bug 或架构问题
4. 任何对用户决策有重要影响的信息

**汇报格式**：
```
## 问题/发现标题

### 问题描述
<清晰描述>

### 影响分析
<分析影响>

### 建议方案
<可能的解决方案>

### 用户决定
<等待用户选择>
```

### 3.3 禁止行为

- ❌ 不经用户确认就做重大架构决策
- ❌ 不询问就删除大量代码
- ❌ 不验证就假设代码没问题
- ❌ 忽略用户的明确指示

## 四、用户工作习惯

### 4.1 决策模式

**快速决策**：
- 用户会在简短分析后给出明确指示
- 通常选择"最小可行方案"或"立即执行"
- 很少需要长时间讨论

**指令类型**：
- "方案B" = 立即执行第二个方案
- "停" = 停止当前工作，等待指示
- "继续" = 按原计划进行
- "方案X" = 执行指定方案

### 4.2 代码查看习惯

- 经常打开文件查看（第1行开始）
- 关注架构层面问题，不只看具体实现
- 对"历史遗留"、"妥协性代码"非常敏感

### 4.3 任务管理

- 使用 TodoWrite 跟踪任务状态
- 每个 Phase 有独立的任务列表
- 完成后汇报整个 Phase 的完成情况

## 五、代码风格规范

### 5.1 命名规范

**类名**：
- PascalCase：`IbPlugin`, `ModuleLoader`
- 不使用前缀/后缀标记类型

**函数/方法名**：
- snake_case：`get_vtable()`, `create_implementation()`
- 方法名要有明确语义

**变量名**：
- snake_case：`impl`, `raw_vtable`
- 避免单字母（循环变量除外）

**常量**：
- UPPER_SNAKE_CASE：`MAX_DEPTH`, `IES_VERSION`

### 5.2 导入规范

**分组顺序**：
1. 标准库
2. 第三方库
3. 项目内部

**相对 vs 绝对**：
- 使用绝对导入：`from core.extension import ibcext`
- 避免过多相对导入

### 5.3 文档字符串

**模块级**：
```python
"""
[IES 2.2] 模块描述

简短说明模块职责。
"""
```

**类级**：
```python
class IbPlugin:
    """
    [IES 2.2] 类描述。

    详细说明类的作用和使用方式。
    """
```

**方法级**：
```python
def get_vtable(self) -> Dict[str, Callable]:
    """
    [IES 2.2] 方法描述。

    Args:
        param1: 参数说明
    Returns:
        返回值说明
    """
```

### 5.4 注释规范

**不要添加无意义注释**：
```python
# 错误：明显的注释
x = x + 1  # x 加 1

# 正确：有意义的注释
x = x + 1  # 绕过边界检查
```

**用注释说明为什么，不是做什么**：
```python
# 正确：解释原因
# 需要在 setup 前注入，因为插件根据能力动态决定虚表
self._setup_implementation(implementation, context, capabilities)
```

## 六、用户厌恶的实现

### 6.1 历史遗留代码

**最讨厌**：
- 兼容性回退代码（`if fallback: ... else: ...`）
- 妥协性设计（"先用着，以后再改"）
- 装饰器/接口/实现混淆

**处理方式**：
- 立即删除旧协议兼容代码
- 不保留"以防万一"的回退
- 要求彻底重构，不接受补丁

### 6.2 不完整的实现

**不接受**：
- TODO 注释
- "最小版本先用"但不说明后续计划
- 半完成的架构设计

**正确做法**：
- 完成就是完成，没完成就是没完成
- 明确标注"最小版本"并说明限制
- 后续工作在 PENDING_TASKS.md 记录

### 6.3 冗余代码

**删除**：
- 未使用的导入
- 空行填充的代码
- 重复的实现

**保留**：
- 必要的架构抽象
- 有明确作用的中间变量
- 清晰的代码组织

## 七、架构决策原则

### 7.1 分层原则

**严格分层**：
- core/ = 解释器核心，不含业务逻辑
- ibc_modules/ = 第一方插件
- plugins/ = 第三方插件

**依赖方向**：
- 业务 → 核心（单向）
- 插件 → 核心（单向）
- 核心不依赖业务

### 7.2 插件分类

**零侵入插件**：
- 不继承任何核心类
- 纯 Python 实现
- 通过 `_spec.py` 的 `__ibcext_metadata__()` 和 `__ibcext_vtable__()` 注册

**核心级插件**：
- 继承 `IbPlugin` 或 `ILLMProvider`
- 需要解释器运行时绑定
- 放在 `ibc_modules/` 目录

### 7.3 协议优先

**IES 2.2 协议**：
- `__ibcext_metadata__()` = 插件元数据
- `__ibcext_vtable__()` = 方法映射表
- `create_implementation()` = 工厂函数（可选）

**禁止**：
- ❌ 装饰器体系（`@method`, `@module`）
- ❌ 侵入式注册
- ❌ 硬编码依赖

## 八、任务执行流程

### 8.1 接收任务

1. 理解任务目标
2. 评估风险和影响
3. 如有疑问立即询问
4. 制定执行计划（用 TodoWrite）

### 8.2 执行任务

1. 小步迭代，每步验证
2. 重要修改后 `git status`
3. 完成一个逻辑单元后 commit
4. 如遇问题立即汇报

### 8.3 完成任务

1. 最终 `git status` 确认
2. `git log --oneline -5` 确认历史
3. 向用户汇报完成情况
4. 更新相关文档（NEXT_STEPS_PLAN.md / PENDING_TASKS.md）

## 九、文档维护

### 9.1 NEXT_STEPS_PLAN.md

- 记录 Phase 任务的完成状态
- 每次 Phase 完成后更新
- V3.2 → V4.0 表示重大架构变更

### 9.2 PENDING_TASKS.md

- 记录被搁置的任务
- 说明搁置原因
- 提供未来实现方向
- 新发现的任务及时添加

### 9.3 commit message

- 中文描述
- 简洁明了
- 说明做了什么，不说明为什么

## 十、关键经验总结

### 10.1 成功经验

1. **小步快走**：每次只做一件事
2. **立即验证**：修改后马上 git status
3. **彻底清理**：不保留历史垃圾
4. **主动汇报**：问题立即说，不累积

### 10.2 教训

1. **不要假设**：遇到不确定要问
2. **不要堆积**：历史问题立即清理
3. **不要妥协**：架构问题不能用补丁
4. **不要猜测**：用户意图不明确时停止并询问
